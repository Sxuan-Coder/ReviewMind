# AGENTS.md

本文件用于约束本仓库中所有AI Agent、自动化工具和人类开发者的行为。

所有 Agent 在读取、修改、生成、审查、提交本仓库内容前，必须遵守本文件。

---

## 1. 项目评审重点

评审维度如下：

### 1.1 作品完整度与创新性：40%

重点关注：

- 产品设计是否合理
- 功能是否完整
- 交互是否流畅
- 是否具有新颖性和创意
- 是否符合所选议题方向

### 1.2 开发过程与质量：40%

重点关注：

- 架构是否清晰合理
- 代码是否健壮
- 逻辑是否正确
- 代码规范性、可读性、可维护性
- PR 数量与质量
- commit 分布是否合理
- 主分支是否持续保持可运行状态

### 1.3 演示与表达：20%

重点关注：

- Demo 视频是否清晰完整
- 是否有声音讲解
- 是否覆盖核心模块
- 是否清晰表达作品功能和效果

## 2. 禁止规则

- 伪造 commit 时间
- 伪造 PR 过程
- 伪造测试结果
- 生成与实际代码变更不一致的 PR 描述
- 提交参赛者未理解、未审查、未测试的大段代码
- 禁止提交ClaudeCode和其他工具PR的署名
- 禁止将 AI 工具署名为项目贡献者
- 如工具自动生成 `Co-authored-by`、`Generated by` 等署名信息，应在提交前人工检查

## 3. 作品有效性要求

### 3.1 必须符合议题方向

作品必须符合所选比赛议题方向。

禁止提交与议题方向明显无关的项目。

### 3.2 严禁抄袭

禁止：

- 抄袭他人代码
- 未声明复制开源项目
- 未声明使用第三方模板
- 未声明复用自己过去的代码
- 使用侵犯知识产权的内容

## 4. 开发过程要求

### 4.1 必须持续交付

从议题发布之日起，开发周期内必须保持持续的 PR 和 commit 记录。

禁止临近截止时间一次性导入所有代码。

### 4.2 严禁临尾突击提交

禁止以下行为：

```bash
git add .
git commit -m "init"
git commit -m "final"
git commit -m "complete project"
git commit -m "all features"
```

除非该提交确实只包含对应的小范围修改。

### 4.3 commit 时间要求

所有 commit 时间戳必须落在所选批次的开始时间与截止时间之内。

本项目当前批次时间范围为：

- 开始时间：2026-05-29 00:00:00
- 截止时间：2026-05-31 23:59:59

禁止为了规避比赛审查而伪造 commit 时间。

### 4.4 禁止重写历史规避审查

Agent 不得主动执行以下操作：

```bash
git rebase -i
git filter-branch
git filter-repo
git reset --hard
git push --force
```

除非开发者明确授权，且必须提前说明风险。

## 5. PR 提交规范

### 5.1 必须基于 PR 添加新功能

新功能、重要修改、结构调整、文档更新，应通过 Pull Request 合入。

禁止长期直接在主分支上开发大功能。

推荐流程：

```bash
git checkout -b feature/<short-feature-name>
git add <files>
git commit -m "feat: add <specific feature>"
git push origin feature/<short-feature-name>
# 创建 PR
```

### 5.2 每个 PR 只做一件事

每个 PR 只实现或修改单一功能。

要求：

- 每个 PR 只做一件事
- 鼓励尽可能小、粒度尽可能细的 PR
- 大功能应拆分为多个独立 PR 分步提交
- 每个 PR 合并后，主分支代码必须保持可运行状态

允许的 PR 示例：

```text
feat: add repository analyzer module
feat: add PR description quality checker
test: add parser unit tests
docs: add backend startup guide
fix: handle empty commit list
```

禁止的 PR 示例：

```text
feat: add all features
update project
final version
fix everything
massive refactor and UI and backend and docs
```

### 5.3 大功能必须拆分

例如，一个完整的 AI PR Review 系统不应一次性提交，应拆分为：

```text
1. feat: initialize project structure
2. feat: add repository metadata parser
3. feat: add commit history analyzer
4. feat: add PR description quality checker
5. feat: add code quality rule engine
6. feat: add review report generator
7. feat: add frontend review dashboard
8. feat: connect frontend to backend review API
9. test: add review engine unit tests
10. docs: add usage guide and demo link
```

禁止把完整系统在一个 PR 中一次性完成。

## 6. PR 标题与描述要求

每个 PR 的标题与描述必须清晰完整，并且与实际代码变更一致。

PR 内容必须包含：

- 标题
- 功能描述
- 实现思路
- 测试方式

### 6.1 标题要求

标题必须用一句话说明本 PR 新增或修改了什么。

示例：

```text
feat(auth): 实现手机号验证码登录
```

### 6.2 功能描述要求

必须说明该功能的作用与使用方式。

例如：

````md
## 功能描述

本 PR 新增 commit history analyzer，用于读取仓库提交记录，分析 commit 时间分布、提交信息规范性以及是否存在集中提交风险。

使用方式：

```bash
npm run analyze:commits
```
````

### 6.3 实现思路要求

必须简要说明技术选型或核心实现逻辑。

例如：

```md
## 实现思路

本 PR 通过 Git 命令读取 commit log，并将提交时间、作者、message 解析为结构化数据。
随后根据规则检查：

- commit 是否集中在短时间内
- commit message 是否过于空泛
```

### 6.4 测试方式要求

必须说明如何验证该功能正常运行。

例如：

````md
## 测试方式

1. 执行：

```bash
npm test
npm run analyze:commits
```

2. 使用包含正常 commit 和异常 commit 的测试仓库进行验证。
3. 确认输出报告中能正确识别集中提交和空泛 commit message。
````

### 6.5 PR 合并后要求

团队开发流程：

```text
main（最终稳定版 → 永远可运行、可演示）
↑
dev（开发集成版 → 功能合并、测试）
↑
feat/xxx（你写功能）
feat/yyy（队友写功能）
```

PR 合并后，主分支代码必须保持可运行状态，必须先测试通过后才能合并入主分支。

评委在任意时间查看仓库时，应能复现演示效果。

## 8. commit 规范

### 8.1 commit 必须小而清晰

每个 commit 应对应一个清晰修改点。

### 8.2 禁止空泛 commit

禁止以下 commit 信息：

```text
update
fix
final
done
wip
all
提交
修改
完善
```

除非信息中明确说明具体修改内容。

### 8.3 commit 内容必须真实

commit message 必须准确反映代码变更。

禁止：

- 一个 commit 同时塞入多个不相关功能
- 用 `docs` 提交代码功能
- 用 `fix` 提交大规模重构
- 用 `feat` 提交与功能无关的文件
- 为了制造提交数量拆分无意义 commit

---

## 9. 多 Agent 协作规则

如果使用多个 Agent，例如 Claude Code等，必须遵守以下规则。

### 9.1 禁止多个 Agent 无约束修改同一范围

禁止多个 Agent 同时大规模修改同一批文件。

推荐做法：

- 每次只让一个 Agent 修改一个明确任务
- 修改前明确目标和文件范围
- 修改后人工 review
- 再交给其他 Agent 做审查或测试建议

### 9.2 多 Agent 输出必须人工确认

Agent 输出不得直接合并。

必须由参赛者人工判断后选择性采纳。

## 10. Agent 可执行操作范围

### 10.1 Agent 修改代码的限制

Agent 修改代码时必须遵守：

- 每次只修改与当前任务相关的文件
- 不得无关格式化整个仓库
- 不得批量重写项目结构
- 不得删除已有测试
- 不得删除已有文档说明
- 不得引入未声明依赖
- 不得提交无法运行的主分支代码

### 10.2 Agent 不得执行的高风险操作

Agent 不得主动执行：

```bash
git push --force
git reset --hard
git clean -fd
rm -rf .git
rm -rf *
git filter-branch
git filter-repo
```

除非开发者明确授权，且操作前必须解释风险。

## 11. 第三方依赖与原创性要求

### 11.1 新增依赖必须登记

如果引入第三方库、框架、SDK、模板、UI 库、模型、API，必须在 README 中说明。

示例：

```md
## 依赖

| Name       | Purpose            | Version | Original Work Boundary                         |
| ---------- | ------------------ | ------- | ---------------------------------------------- |
| React      | Frontend framework | x.x.x   | UI and business logic are original             |
| Express    | Backend server     | x.x.x   | API design and review logic are original       |
| OpenAI SDK | Model API client   | x.x.x   | Prompt design and review workflow are original |
```

### 11.2 必须说明原创功能边界

README 中必须说明：

- 哪些部分是原创实现
- 哪些部分依赖第三方库
- 是否参考公开文档或示例
- 是否复用过往代码

### 11.3 复用旧代码必须声明

如果复用自己过去的代码，必须在 README 中说明。

未声明复用旧代码可能导致作品无效。

```md
## 功能列表

说明 frontend、backend、agent、database、model、API 等模块。

## 安装与运行

提供完整可复现的启动步骤。

## 第三方依赖说明

## 原创性说明
```

## 13. Demo 视频要求

### 13.1 必须有声音讲解

Demo 视频必须使用声音完整讲解。

不能只录屏不讲解。

### 13.2 必须展示主要功能和效果

Demo 视频必须展示作品主要功能和实际运行效果，并覆盖核心模块。

至少应包含：

- 项目背景和目标
- 核心功能展示
- 关键交互流程
- 主要技术亮点
- 实际运行结果
- 如何复现或启动项目

### 13.3 必须上传至可访问外部平台

Demo 视频必须上传至可访问的外部平台，例如：

- bilibili
- 云盘
- 其他可公开视频平台或文件平台

### 13.4 README 必须放置视频链接

Demo 视频链接必须放到代码仓库的 README 中。

要求：

- 链接必须可访问
- 视频必须可播放
- 链接必须放到 README 显眼位置
- 不要藏在文件夹或子文件夹内
- 不要只放在 issue、PR、release 或其他不明显位置

### 13.5 Demo 内容必须真实

禁止在 demo 中展示：

- 未实现功能
- 无法运行的功能
- mock 成真实能力的功能
- 与代码仓库不一致的效果
- 只在特殊本地环境可运行但 README 未说明的能力
- 截止后无法复现的能力

Agent 可以帮助生成 demo 脚本，但不得编造不存在的功能。

## 14. 主分支可运行要求

PR 合并后，主分支代码必须保持可运行状态。

评委在任意时间查看时，应能复现演示效果。

每次合并前，应根据项目实际情况执行：

```bash
pnpm install
pnpm run lint
pnpm test
pnpm run build
pnpm run dev
```

或项目 README 中声明的等效命令。

如果某些命令暂时不可用，必须在 PR 描述中说明原因。

---

## 15. 测试要求

新增核心逻辑时，必须优先补充测试。

推荐测试范围：

- 正常输入
- 空输入
- 异常输入
- 边界条件
- 第三方 API 失败
- 文件不存在
- 配置缺失
- 权限不足
- 网络失败

Agent 生成测试时，必须保证测试真实可运行。

禁止只写表面测试、伪测试或无法执行的测试。

---

## 16. 安全与密钥管理

禁止提交：

- API Key
- Token
- Cookie
- 私钥
- 数据库密码
- 个人账号凭证
- 真实用户隐私数据

必须使用：

```text
.env
.env.local
.env.example
```

`.env.example` 只能包含示例值。

示例：

```env
OPENAI_API_KEY=your_api_key_here
DATABASE_URL=your_database_url_here
```

```md
- [ ] 是否存在无关文件变更？
- [ ] 是否引入第三方依赖？
- [ ] 新依赖是否写入 README？

- [ ] README 是否需要更新？

```

## 19. 禁止事项总结

任何 Agent 或开发者不得：

- 伪造测试结果
- 使用空白或虚假 PR 描述
- 提交无法运行的主分支
- 提交密钥或隐私数据
- 为了制造开发痕迹拆分无意义 commit
- 让多个 Agent 同时无约束修改同一代码区域

## 20. 推荐开发节奏

推荐按照以下节奏进行：

```text
第 1 阶段：项目初始化

- 创建新仓库
- 编写 README 初稿
- 确定技术栈
- 提交基础目录结构
- 建立 PR 模板

第 2 阶段：核心能力拆分

- 每个核心模块单独 PR
- 每个 PR 保持小粒度
- 每个 PR 合并后主分支可运行
- 添加基础测试

第 3 阶段：功能完善

- 补充交互
- 增强异常处理
- 完善测试
- 优化代码结构

第 4 阶段：演示准备

- 更新 README
- 录制 demo 视频
- 上传 demo 视频到可访问平台
- 将 demo 链接放到 README 显眼位置
- 检查依赖和原创性说明

第 5 阶段：最终检查

- 检查 PR 质量
- 检查 commit 分布
- 检查主分支可运行
- 检查 README 完整性
- 检查 demo 视频可播放
- 检查是否存在未声明依赖、旧代码
```

## 21. 仓库结构要求

如果项目包含多个独立模块，应放在同一个仓库的不同子目录中。

推荐结构：

```text
.
├── frontend/
├── backend/
├── agents/
├── docs/
├── tests/
├── scripts/
├── README.md
├── AGENTS.md
└── .github/
    └── pull_request_template.md
```

要求：

- frontend 和 backend 分别说明启动方式
- agents 目录说明 Agent 工作流或提示词边界
- docs 目录可放设计文档
- tests 目录放测试代码
- scripts 目录放辅助脚本
- README 必须能指导评委完整运行项目

多人组队：

```md
## 团队贡献
```

## 23. 优先级规则

如果其他文档、Agent 默认规则或工具建议与本文件冲突，以更严格、更符合比赛规则的要求为准。

优先级如下：

```text
比赛官方规则 > AGENTS.md > README > PR 模板 > Agent 默认行为 > 临时建议
```

任何不确定情况，必须选择更透明、更保守的做法。

## 24. 最终声明

所有提交、PR、README和最终作品必须真实、清晰、可运行、可审查、可复现。
